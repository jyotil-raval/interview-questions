## Hooks › useState

- **Question:** How does `useState` work under the hood in terms of call order and state persistence, and what are the common bugs that emerge from misunderstanding it?

- **Answer:**
  - `useState` hooks into React's Fiber node for the current component — each call appends a state cell to a linked list stored on the fiber, identified by call order, not variable name.
  - This is why hooks cannot be called conditionally — the call order must be identical on every render for React to associate each call with its corresponding state cell.
  - State updates via the setter are asynchronous and batched — React doesn't immediately mutate the state value; it schedules a re-render and the new value is available in the next render cycle.
  - The stale closure bug: if you capture state in a closure (setTimeout, event handler) without using the functional updater form, you read the value from the render at closure creation time, not the current value.
  - The functional updater form (`setState(prev => prev + 1)`) is the correct pattern for updates that depend on previous state — it receives the latest state at dispatch time, not closure time.

- **Example:**

  ```jsx
  // Stale closure bug — classic 3am incident
  const Counter = () => {
    const [count, setCount] = useState(0);

    const handleClick = () => {
      setTimeout(() => {
        setCount(count + 1); // BUG: count is stale — always reads initial 0
      }, 1000);
    };

    // Fix: functional updater form
    const handleClickFixed = () => {
      setTimeout(() => {
        setCount((prev) => prev + 1); // correct: receives latest count
      }, 1000);
    };
  };
  ```

- **Technical Terms to Include:** Fiber node, linked list, call order invariant, stale closure, functional updater, batched updates, re-render cycle, state cell.

- **Gotcha:** "What happens if you call `setState` with the same value?" → React bails out of the re-render (Object.is comparison). But if you mutate an object in place and call setState with the same reference, React won't re-render — a common source of bugs when developers mutate state directly.

- **Follow-Up:** "What changed with state batching in React 18?" → Prior to React 18, batching only happened in React event handlers. React 18 introduced automatic batching across all contexts — setTimeout, Promises, native events. They ask to test React 18 awareness.

- **Conclusion:** `useState` works by call-order indexing into a per-component linked list — the stale closure pattern and direct mutation are the two failure modes that catch developers who don't internalise this.

## Hooks › useEffect

- **Question:** Explain `useEffect`'s execution model, the dependency array contract, and the three categories of bugs it produces when misused.

- **Answer:**
  - `useEffect` runs after the browser has painted — it's asynchronous relative to rendering. The cleanup function returned from the effect runs before the next effect execution and on unmount.
  - The dependency array tells React when to re-run the effect: empty array = run once after mount; no array = run after every render; populated array = run when any listed value changes.
  - Bug category 1 — Missing dependency (stale effect): the effect captures a value at mount time but that value changes later — the effect never sees the update. ESLint `exhaustive-deps` catches this.
  - Bug category 2 — Oversubscribed dependency (infinite loop): an effect creates a new object/function on each render and that object is listed as a dependency — triggers effect → state update → re-render → new object → effect again.
  - Bug category 3 — Missing cleanup (memory leak / race condition): an effect starts an async operation; the component unmounts before it resolves; the async callback still updates state on an unmounted component, or a newer fetch response is overwritten by a stale one.

- **Example:**

  ```jsx
  // Bug 3: race condition — Apple Offers API fetch without cleanup
  useEffect(() => {
    let cancelled = false;
    fetchOffers(deviceId).then((data) => {
      if (!cancelled) setOffers(data); // guard against stale response
    });
    return () => {
      cancelled = true;
    }; // cleanup: cancel on unmount/re-run
  }, [deviceId]);

  // Bug 2: infinite loop — new object reference each render
  useEffect(() => {
    fetchData(filters); // filters is {} created inline in render
  }, [filters]); // new reference every render = infinite loop
  ```

- **Technical Terms to Include:** effect cleanup, dependency array contract, stale closure, exhaustive-deps, race condition, AbortController, referential equality, synchronisation model.

- **Gotcha:** "`useEffect` is for side effects — but what's a side effect in React's definition?" → Any operation that reaches outside the render function: fetching data, DOM manipulation, subscriptions, timers, logging. They want precision, not a vague answer.

- **Follow-Up:** "When should you use `useLayoutEffect` instead of `useEffect`?" → When you need to read layout/DOM measurements and synchronously apply changes before the browser paints — otherwise there's a flash. They're testing render pipeline depth.

- **Conclusion:** `useEffect` is a synchronisation primitive, not a lifecycle hook — every misuse traces back to misunderstanding the dependency array as a contract, not a performance hint.

## Component Lifecycle › Class vs Hooks Mental Model

- **Question:** Map the class component lifecycle methods to their Hook equivalents. Where do the mental models diverge, and why does that divergence matter?

- **Answer:**
  - `componentDidMount` → `useEffect(() => {}, [])`: runs once after first render.
  - `componentDidUpdate` → `useEffect(() => {}, [dep])`: runs after render when `dep` changes.
  - `componentWillUnmount` → cleanup function returned from `useEffect`.
  - `shouldComponentUpdate` → `React.memo` for functional components; `useMemo`/`useCallback` for internal computations.
  - The critical divergence: class lifecycle methods are tied to a component's existence (mount/update/unmount). `useEffect` is tied to synchronisation — "run this effect when these values change." This is a fundamentally different mental model.

- **Example:**

  ```jsx
  // Class: lifecycle-oriented thinking
  componentDidMount() { this.fetchOffers(); }
  componentDidUpdate(prevProps) {
    if (prevProps.deviceId !== this.props.deviceId) this.fetchOffers();
  }
  componentWillUnmount() { this.cleanup(); }

  // Hook: synchronisation-oriented thinking
  useEffect(() => {
    fetchOffers(deviceId);
    return () => cleanup();
  }, [deviceId]); // "keep this in sync with deviceId"
  ```

- **Technical Terms to Include:** mount/update/unmount lifecycle, synchronisation model, `React.memo`, `PureComponent`, `getDerivedStateFromProps`, `useLayoutEffect`, fiber reconciler.

- **Gotcha:** "Is there a Hook equivalent of `componentDidCatch`?" → No — error boundaries still require class components as of React 18. React's `use()` API and upcoming concurrent features may change this, but it's a known gap.

- **Follow-Up:** "Why did React deprecate `componentWillReceiveProps` and `componentWillUpdate`?" → They were called during the render phase (before commit), making them unsafe in concurrent mode where renders can be interrupted and retried. They probe Concurrent Mode awareness.

- **Conclusion:** Migrating from class lifecycle to Hooks requires a mental model shift — from "what happens at lifecycle stages" to "what values should this effect stay synchronised with."

## Props vs State › Ownership and Mutability

- **Question:** Define props and state precisely. What determines which one a piece of data should be, and what are the signs a developer has chosen wrong?

- **Answer:**
  - Props are inputs to a component — immutable from the component's perspective, owned and controlled by the parent. State is data owned and managed by the component itself, mutable via its setter.
  - Decision rule: if the data is passed in from outside → props. If the component generates or owns the data → state. If it can be derived from existing props or state → neither (compute it in the render function).
  - Sign of wrong choice #1 — Copying props into state: `const [value, setValue] = useState(props.value)`. The state copy diverges from the prop immediately — the component now has two sources of truth for the same data.
  - Sign of wrong choice #2 — Lifting state too late: two sibling components needing the same data, each maintaining their own local copy — they drift. Solution: lift to the nearest common ancestor.
  - Sign of wrong choice #3 — Derived state as state: storing `const [fullName, setFullName] = useState(firstName + lastName)` — now you must remember to update `fullName` every time either changes. Compute it inline instead.

- **Example:**

  ```jsx
  // Wrong: copying prop into state creates divergence
  const OfferBadge = ({ discount }) => {
    const [localDiscount, setLocalDiscount] = useState(discount); // stale after prop update
  };

  // Correct: derive directly, no stale copy
  const OfferBadge = ({ discount }) => {
    const label = discount > 20 ? 'HOT' : 'DEAL'; // derived, never stale
    return (
      <span>
        {label}: {discount}%
      </span>
    );
  };
  ```

- **Technical Terms to Include:** single source of truth, lifting state, derived state, controlled component, prop drilling, ownership model, `getDerivedStateFromProps`.

- **Gotcha:** "When IS it valid to initialise state from props?" → When the prop is used only as an initial/seed value and the component intentionally diverges from the parent — e.g., a text field that starts with a default value. The key word is "initialiser," not "sync target."

- **Follow-Up:** "What is the `key` trick for resetting component state?" → Setting `key={someId}` on a component forces React to unmount and remount it when the key changes — cleanly resetting all internal state. They test whether you know this escape hatch.

- **Conclusion:** Props are contract inputs owned by the parent; state is internal memory owned by the component — violating this ownership model produces two-source-of-truth bugs that are painful to debug at scale.

## Keys in Lists › Identity and Reconciliation

- **Question:** Why is `key` a critical architectural concern in list rendering, and what failure modes emerge from incorrect key usage?

- **Answer:**
  - `key` is React's mechanism to maintain stable component identity across renders — it tells the reconciler which item is which between the old and new Virtual DOM trees.
  - Without keys, React maps list items by position — if the list reorders, React reuses DOM nodes in their original positions, potentially applying state from item A to what is now item B.
  - Using array index as key breaks on: deletion (all subsequent indexes shift), insertion at the beginning (every item gets a new index), and reordering.
  - Keys must be stable (not regenerated each render), unique within the list (not globally), and predictable — using `Math.random()` as a key unmounts and remounts every item on every render, destroying all child state and focus.
  - In a PayPal payment method list, incorrect keys caused a previously entered CVV field to appear populated in the wrong card slot after a reorder — a security-relevant UI bug.

- **Example:**

  ```jsx
  // Bad: index key — breaks on reorder/delete
  paymentMethods.map((method, i) => <PaymentCard key={i} method={method} />);

  // Bad: random key — unmounts every card on every render
  paymentMethods.map((method) => <PaymentCard key={Math.random()} method={method} />);

  // Correct: stable, unique business ID
  paymentMethods.map((method) => <PaymentCard key={method.instrumentId} method={method} />);
  ```

- **Technical Terms to Include:** stable identity, reconciler, fiber node, keyed reconciliation, DOM node reuse, index-as-key footgun, unmount/remount cycle, list diffing.

- **Gotcha:** "Keys are only for lists, right?" → No. The key prop works on any element — it's a general identity hint to the reconciler. You can use `key={someId}` on a single component to force a full reset when an ID changes. This is the "key reset trick."

- **Follow-Up:** "Can two sibling lists share the same key values?" → Yes — keys are only required to be unique among siblings within the same parent. They test whether you know the scope of key uniqueness.

- **Conclusion:** Key is not a performance hint — it is a correctness contract; incorrect keys silently corrupt component state in ways that surface as intermittent UI bugs under specific interaction patterns.
