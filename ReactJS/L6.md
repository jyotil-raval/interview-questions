## Expert › React Fiber Architecture

- **Question:** Explain React Fiber's architecture. What fundamental limitation of the previous stack reconciler did it solve, and what capabilities did it unlock?

- **Answer:**
  - The pre-Fiber "Stack Reconciler" processed the component tree in a single synchronous call stack — once started, reconciliation ran to completion and couldn't be interrupted. Long trees caused frame drops because the main thread was blocked.
  - Fiber reimplements the reconciler as a linked list of "fiber nodes" — each node represents a unit of work (a component). Work can be paused, resumed, prioritised, and discarded between fibers.
  - A fiber node contains: the component type, its props/state, a reference to the parent/child/sibling fiber, and work-in-progress state. React maintains two trees: the "current" tree (what's on screen) and the "work-in-progress" tree (what's being computed).
  - This interruptibility unlocks: (1) Concurrent Mode — high-priority updates (user input) can interrupt low-priority background renders; (2) Suspense — a fiber can pause and wait for a Promise; (3) `startTransition` — non-urgent state updates can yield to urgent ones; (4) `useTransition`/`useDeferredValue` — explicit priority markers for developers.
  - The commit phase remains synchronous — once React determines what changes to apply, it applies them atomically to avoid inconsistent intermediate UI states.

- **Example:**

  ```
  // Fiber tree structure (conceptual)
  App fiber {
    type: App,
    child: → Header fiber {
      type: Header,
      sibling: → OffersGrid fiber {
        type: OffersGrid,
        child: → OfferCard fiber [0] {
          type: OfferCard,
          sibling: → OfferCard fiber [1] { ... }
        }
      }
    },
    return: null (root)
  }
  // React traverses: DFS via child → sibling → return pointers
  // Can pause between any two fibers and resume later
  ```

- **Technical Terms to Include:** Fiber, work unit, linked list, work-in-progress tree, current tree, interruptible reconciliation, commit phase, Concurrent Mode, priority lanes, `startTransition`, `useDeferredValue`.

- **Gotcha:** "Does Fiber mean React renders asynchronously?" → Fiber makes reconciliation interruptible, but rendering can still be synchronous (legacy mode). Concurrent Mode (opt-in in React 18 via `createRoot`) is what actually enables asynchronous rendering with priority scheduling.

- **Follow-Up:** "What are priority lanes in React 18?" → Lanes is the scheduling model inside Fiber — each update is assigned a priority lane (SyncLane, InputContinuousLane, DefaultLane, TransitionLane, IdleLane). The scheduler processes higher-priority lanes first. They're testing deep internals knowledge.

- **Conclusion:** Fiber is not a performance optimisation of the old reconciler — it's a complete architectural rewrite that transforms React's rendering from a blocking synchronous operation into a scheduled, interruptible, prioritised unit-of-work system.

## Expert › Concurrent Mode and `useTransition`

- **Question:** What is Concurrent Mode in React 18, how does `useTransition` expose it to developers, and what failure modes does it introduce?

- **Answer:**
  - Concurrent Mode enables React to prepare multiple versions of the UI simultaneously — it can start rendering a new state while keeping the previous state visible, abandon a render in progress if a higher-priority update arrives, and schedule renders based on urgency.
  - Enabled by `ReactDOM.createRoot()` instead of `ReactDOM.render()` in React 18. Once enabled, React automatically applies concurrency for Suspense and transitions.
  - `useTransition` returns `[isPending, startTransition]`. State updates wrapped in `startTransition` are marked as non-urgent. React will show the current UI while preparing the transition in the background — no jarring flash to a Suspense fallback.
  - `useDeferredValue` is the value-based equivalent — it accepts a value and returns a deferred version that updates with lower priority. Useful for input fields where you want the field to update immediately but an expensive filtered list to update with delay.
  - Failure mode 1 — Tearing: in Concurrent Mode, React can read state that's being updated mid-render. External stores (non-React state) that aren't subscribed via `useSyncExternalStore` can expose inconsistent values in the same render pass — a visual tear.

- **Example:**

  ```tsx
  // useTransition: keep current UI visible while filtering large dataset
  const OffersSearch = ({ offers }) => {
    const [query, setQuery] = useState('');
    const [filteredOffers, setFilteredOffers] = useState(offers);
    const [isPending, startTransition] = useTransition();

    const handleSearch = (e) => {
      setQuery(e.target.value); // urgent: input updates immediately

      startTransition(() => {
        // non-urgent: filtering 10k offers happens in background
        setFilteredOffers(offers.filter((o) => o.title.includes(e.target.value)));
      });
    };

    return (
      <>
        <input value={query} onChange={handleSearch} />
        {isPending && <Spinner />}
        <OfferList offers={filteredOffers} />
      </>
    );
  };
  ```

- **Technical Terms to Include:** Concurrent Mode, `createRoot`, `useTransition`, `useDeferredValue`, `startTransition`, tearing, `useSyncExternalStore`, priority scheduling, deferred render, lane model.

- **Gotcha:** "What is tearing and why does it only appear in Concurrent Mode?" → Tearing is when different components in the same render read different values from an external store — because Concurrent Mode can interrupt and resume renders, the store value can change between reads. In synchronous rendering, all reads happen atomically. `useSyncExternalStore` solves this.

- **Follow-Up:** "Which third-party state managers are Concurrent Mode safe out of the box?" → Redux (since v8 with `useSyncExternalStore`), Zustand (v4+), Jotai. MobX had tearing issues in early Concurrent Mode — they've since patched. Any store that doesn't use `useSyncExternalStore` is potentially unsafe.

- **Conclusion:** Concurrent Mode makes React's rendering model non-deterministic from the perspective of external stores — adopting it requires auditing every external state subscription to ensure it uses `useSyncExternalStore` or a patched library implementation.

## Expert › Security (XSS in React)

- **Question:** How does React protect against XSS by default, where does that protection break down, and what patterns introduce XSS vulnerabilities in a React application?

- **Answer:**
  - React's JSX expression interpolation (`{value}`) automatically escapes HTML entities — `<`, `>`, `"`, `'`, `&` are converted to their HTML entity equivalents. A user value containing `<script>` renders as literal text, not executable HTML.
  - This protection only applies to JSX expressions. It breaks down at three escape hatches: (1) `dangerouslySetInnerHTML={{ __html: userContent }}` — explicitly bypasses escaping; (2) `javascript:` URLs in `href`/`src` attributes — JSX escapes the string but the browser interprets the protocol; (3) `ref`-based imperative DOM manipulation that sets `innerHTML` directly.
  - Pattern that introduces XSS: rendering user-supplied Markdown or rich text via `dangerouslySetInnerHTML` without sanitisation. The content must be sanitised with a library like DOMPurify before being passed to `dangerouslySetInnerHTML`.
  - On PayPal's Checkout, merchant-supplied display names were rendered as text nodes (safe), but a legacy component used `dangerouslySetInnerHTML` for formatted amounts — a code review flag that resulted in mandatory DOMPurify sanitisation being added to the pipeline.
  - Server-side: React's `renderToString` also escapes, but JSON serialised into `<script>` tags (for hydration data) can be attacked via script injection if not properly serialised — use `JSON.stringify` with HTML encoding, or libraries like `serialize-javascript`.

- **Example:**

  ```tsx
  // Safe: React escapes this — renders as text, not HTML
  const userInput = '<script>alert("xss")</script>';
  return <div>{userInput}</div>; // renders literal text ✅

  // Dangerous: bypasses escaping
  return <div dangerouslySetInnerHTML={{ __html: userInput }} />; // XSS ❌

  // Correct: sanitise before dangerouslySetInnerHTML
  import DOMPurify from 'dompurify';
  const sanitised = DOMPurify.sanitize(userInput);
  return <div dangerouslySetInnerHTML={{ __html: sanitised }} />; // ✅

  // javascript: URL — escaping doesn't help here
  const url = 'javascript:alert("xss")';
  return <a href={url}>Click</a>; // browser executes the protocol ❌
  // Fix: validate URL protocol before rendering
  const safeUrl = url.startsWith('http') ? url : '#';
  ```

- **Technical Terms to Include:** XSS (Cross-Site Scripting), `dangerouslySetInnerHTML`, DOMPurify, HTML entity escaping, `javascript:` protocol, Content Security Policy (CSP), serialisation, DOM injection, sanitisation.

- **Gotcha:** "Is React completely XSS-proof by default?" → No. React's JSX escaping handles the most common case, but `dangerouslySetInnerHTML`, `javascript:` URLs, `eval()` in the codebase, and inline event attribute strings are all unprotected. Defence-in-depth requires CSP headers in addition to React's escaping.

- **Follow-Up:** "What is Content Security Policy and how does it complement React's XSS protection?" → CSP is an HTTP header that restricts which scripts, styles, and resources the browser will execute — even if an attacker injects a `<script>` tag, CSP prevents it from running unless it matches the policy. It's the last line of defence that React's escaping doesn't provide.

- **Conclusion:** React's JSX escaping is a strong first line of defence against XSS, but `dangerouslySetInnerHTML`, `javascript:` URLs, and serialisation of hydration data are the three specific surfaces where XSS vulnerabilities in React applications are actually found.

## Expert › Accessibility (a11y) in React

- **Question:** What are the key accessibility requirements a React Tech Lead should enforce across a component library, and where do React's patterns create accessibility anti-patterns?

- **Answer:**
  - The WCAG foundation: perceivable (content is available to all senses/assistive tech), operable (all interactions are keyboard-navigable), understandable (predictable behaviour), robust (works with assistive technology like screen readers).
  - React-specific a11y concerns: (1) Focus management — modals, drawers, and dialogs must trap focus when open and restore focus to the trigger on close. React's rendering doesn't do this automatically. (2) Live regions — dynamically updated content (status messages, loading states) must use `aria-live` regions so screen readers announce changes without focus movement. (3) Semantic HTML — JSX makes it easy to build everything with `<div>` and `<span>`, losing the implicit ARIA roles of native elements (`<button>`, `<nav>`, `<main>`).
  - The SPA routing anti-pattern: React Router navigates without a page reload — screen readers don't announce the route change because no document is loaded. You must programmatically announce route changes via an `aria-live` region or set focus to the new page's `<h1>`.
  - Component library enforcement: ESLint `jsx-a11y` plugin catches common violations at build time. Axe-core integrated into testing (via `@axe-core/react` or Testing Library's `checkAccessibility`) catches runtime violations.

- **Example:**

  ```tsx
  // Focus trap in modal — mandatory for a11y compliance
  const Modal = ({ isOpen, onClose, children }) => {
    const modalRef = useRef(null);

    useEffect(() => {
      if (isOpen) {
        modalRef.current?.focus(); // move focus into modal on open
      }
    }, [isOpen]);

    return isOpen ? (
      <div
        role='dialog'
        aria-modal='true'
        aria-labelledby='modal-title'
        ref={modalRef}
        tabIndex={-1} // makes div focusable
        onKeyDown={(e) => e.key === 'Escape' && onClose()}>
        {children}
      </div>
    ) : null;
  };

  // aria-live region for dynamic updates
  const StatusMessage = ({ message }) => (
    <div aria-live='polite' aria-atomic='true' className='sr-only'>
      {message}
    </div>
  );
  ```

- **Technical Terms to Include:** WCAG, ARIA roles, `aria-live`, `aria-modal`, focus management, focus trap, semantic HTML, `tabIndex`, keyboard navigation, screen reader, `jsx-a11y`, axe-core, route announcement.

- **Gotcha:** "Is `aria-label` always the correct way to add accessible labels?" → No — `aria-labelledby` is preferred when a visible label already exists in the DOM (it references the visible text, reducing duplication). `aria-label` is for when no visible label exists. Using `aria-label` when a visible label exists creates a divergence that confuses assistive tech users.

- **Follow-Up:** "How do you test accessibility in a React CI pipeline?" → `@axe-core/playwright` or Cypress-axe for integration tests. `jest-axe` for unit-level checks in Testing Library tests. Add a threshold: fail the build if any critical/serious axe violations are introduced.

- **Conclusion:** Accessibility in React requires intentional architecture decisions — focus management, live regions, semantic HTML, and SPA route announcements are all responsibilities the framework does not handle automatically, making them Tech Lead enforcement priorities.

## Expert › Testing Strategy for React at Scale

- **Question:** Design the testing pyramid for a large-scale React application. What do you test at each layer, and what are the trade-offs that determine the right balance for a production system?

- **Answer:**
  - The testing pyramid for React: (1) Unit tests — individual functions, custom Hooks (`renderHook`), pure utilities. Fast, cheap, numerous. (2) Component tests — individual components rendered with React Testing Library. Tests user-facing behaviour (what's in the DOM) not implementation (state variable names). (3) Integration tests — multiple components composed together, potentially with mocked API responses. Tests user flows. (4) E2E tests — Playwright/Cypress running real browser against real or staging backend. Tests full user journeys. Slow, expensive, few.
  - React Testing Library's philosophy: test what the user sees and does, not how the component is implemented. `getByRole('button', { name: 'Pay Now' })` is more resilient than `querySelector('.submit-btn')` — the former survives a CSS class refactor; the latter breaks.
  - The anti-pattern: testing implementation details (asserting on state values, calling component methods directly). These tests break on refactoring without any actual user-facing behaviour changing — they produce false negatives and slow down development.
  - Custom Hooks must be tested in isolation — `renderHook` from Testing Library lets you call a Hook outside a component, assert on its return values, and fire state updates.
  - On PayPal's Checkout: unit tests covered the billing calculation logic, component tests covered each payment method's rendering and selection, integration tests covered the checkout flow, and E2E (Playwright) ran the critical "add card → pay" path on every deployment.

- **Example:**

  ```tsx
  // Component test: user-facing behaviour, not implementation
  import { render, screen, userEvent } from '@testing-library/react';

  test('selecting an offer updates the checkout total', async () => {
    render(<OffersGrid offers={mockOffers} onSelect={jest.fn()} />);

    const offerCard = screen.getByRole('button', { name: /20% off/i });
    await userEvent.click(offerCard);

    expect(screen.getByText(/total: \$80/i)).toBeInTheDocument();
    // Asserting on visible output — not on internal state
  });

  // Custom Hook test
  import { renderHook, act } from '@testing-library/react';
  test('useOffers fetches and returns offers', async () => {
    const { result } = renderHook(() => useOffers('device_123'));
    await act(async () => {
      /* wait for useEffect */
    });
    expect(result.current.offers).toHaveLength(3);
    expect(result.current.loading).toBe(false);
  });
  ```

- **Technical Terms to Include:** testing pyramid, React Testing Library, `renderHook`, `userEvent`, `getByRole`, `msw` (Mock Service Worker), snapshot testing, implementation detail testing, E2E, Playwright, Cypress, test coverage vs test quality.

- **Gotcha:** "Should you aim for 100% test coverage?" → Coverage measures which lines execute during tests — it says nothing about whether the important behaviours are tested correctly. 70% coverage with high-value integration tests beats 100% coverage with implementation-detail unit tests. Coverage is a floor metric, not a goal.

- **Follow-Up:** "What is Mock Service Worker (msw) and how does it improve integration tests?" → MSW intercepts network requests at the Service Worker level (browser) or `node-fetch`/`axios` level (Node) — your components and hooks make real fetch calls that are intercepted and mocked, without any code-level mocking. Tests are closer to production behaviour.

- **Conclusion:** The right testing balance for React at scale is heavy on component integration tests (user-behaviour assertions), light on unit tests for pure logic, near-zero snapshot tests, and a small E2E suite covering critical user paths — optimised for catching real regressions, not maximising coverage numbers.
