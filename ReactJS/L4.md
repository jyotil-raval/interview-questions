## Hooks › useReducer

- **Question:** When should you reach for `useReducer` instead of `useState`, and how does it change the state management mental model?

- **Answer:**
  - `useReducer` is appropriate when state transitions are complex: when next state depends on previous state in non-trivial ways, when multiple sub-values are updated together, or when the logic for calculating next state is sophisticated enough to warrant extraction and testing.
  - It implements the Redux pattern locally: `dispatch(action)` → `reducer(state, action)` → new state. The reducer is a pure function — easy to unit test in isolation without rendering a component.
  - The state dispatch API also produces stable references naturally — `dispatch` is guaranteed stable across renders (unlike `setState` callbacks that capture closures), making it safe to pass as a prop to `React.memo`-wrapped children without `useCallback`.
  - Complex form state with interdependencies (field A disables field B, validation depends on multiple fields) is a natural `useReducer` case — managing 5+ related state values with `useState` creates parallel mutation risks.
  - On the Apple Offers checkout flow, offer selection state had enough complexity (selected offer, applied promo, eligibility status, error states) that `useReducer` made the transitions auditable and testable.

- **Example:**

  ```jsx
  // useReducer: offer selection with multiple related state slices
  const offerReducer = (state, action) => {
    switch (action.type) {
      case 'SELECT_OFFER':
        return { ...state, selected: action.offerId, error: null };
      case 'APPLY_PROMO':
        return { ...state, promo: action.code, discount: action.discount };
      case 'FETCH_ERROR':
        return { ...state, error: action.message, selected: null };
      default:
        return state;
    }
  };

  const [offerState, dispatch] = useReducer(offerReducer, {
    selected: null,
    promo: null,
    discount: 0,
    error: null
  });

  dispatch({ type: 'SELECT_OFFER', offerId: 'OFF_12345' });
  ```

- **Technical Terms to Include:** reducer, dispatch, action, pure function, state machine, co-located logic, stable dispatch reference, action type, state transition.

- **Gotcha:** "`useReducer` is just `useState` under the hood — it adds no real value." → Push back: the value is in co-location of transition logic, testability of the reducer in isolation, and the explicit action model that makes state transitions auditable. The implementation being similar is irrelevant.

- **Follow-Up:** "How does `useReducer` + Context compare to Redux?" → It replicates the Redux pattern locally or app-wide via Context. The difference: Redux adds a devtools ecosystem, middleware (Thunk, Saga), serialisable action logging, and time-travel debugging. `useReducer` + Context is sufficient for medium complexity; Redux wins at large scale.

- **Conclusion:** `useReducer` is the right choice when state transitions need to be explicit, testable, and auditable — it shifts the mental model from "mutate this value" to "what event just happened and what state results from it."

## Concurrent Features › Suspense

- **Question:** What is React Suspense, how does it work mechanically, and what is its architectural significance for data fetching?

- **Answer:**
  - Suspense allows components to "suspend" rendering — throw a Promise — while waiting for asynchronous data. The nearest `<Suspense>` boundary catches the thrown Promise and renders its fallback UI until the Promise resolves.
  - Mechanically: when a component that's being rendered throws a Promise, React pauses the render of that subtree, renders the `<Suspense>` fallback, and retries the suspended component when the Promise resolves.
  - Before Suspense, loading states were managed with `isLoading` booleans scattered across component state — each component responsible for its own loading choreography, producing waterfall renders.
  - Suspense enables a "render as you fetch" model — data fetching begins before the component renders (in framework-level route loaders or React Query's prefetching), so the component renders with data already available or suspended.
  - The architectural significance: Suspense moves loading state coordination from component logic to the component tree boundary — closer to the `try/catch` mental model.

- **Example:**

  ```jsx
  // Suspense boundary: fallback shown while OfferDetails suspends
  const OffersPage = () => (
    <Suspense fallback={<OffersSkeleton />}>
      <OfferDetails offerId={selectedId} /> {/* may suspend */}
    </Suspense>
  );

  // With React Query (Suspense mode): component suspends automatically
  const OfferDetails = ({ offerId }) => {
    const { data } = useSuspenseQuery(['offer', offerId], () => fetchOffer(offerId));
    // data is guaranteed defined here — suspension handled at boundary
    return <div>{data.title}</div>;
  };
  ```

- **Technical Terms to Include:** Suspense boundary, thrown Promise, fallback UI, render-as-you-fetch, waterfall vs parallel fetching, `startTransition`, React Server Components, concurrent rendering.

- **Gotcha:** "Can you use Suspense with `useEffect`-based data fetching?" → No. `useEffect` runs after render; by then the component has already rendered with undefined data. Suspense only works with data sources that integrate with the thrown-Promise protocol — React Query, SWR (Suspense mode), Relay, or React's `use()` hook.

- **Follow-Up:** "What is `startTransition` and how does it relate to Suspense?" → `startTransition` marks a state update as non-urgent, allowing React to continue showing the current UI while the new suspended content loads — preventing jarring fallback flashes on navigation. They test Concurrent Mode depth.

- **Conclusion:** Suspense is not a loading state pattern — it is a coordination primitive that relocates async resolution from component logic to the component tree, enabling `try/catch`-style error and loading boundary architecture.

## Advanced Patterns › Error Boundaries

- **Question:** What are Error Boundaries, how do you implement them, and what rendering errors do they NOT catch?

- **Answer:**
  - An Error Boundary is a class component that implements `componentDidCatch` and/or `getDerivedStateFromError` — it catches JavaScript errors thrown during rendering, in lifecycle methods, and in constructors of the component tree below it.
  - `getDerivedStateFromError` is called during rendering to update state to show a fallback UI. `componentDidCatch` is called after rendering for logging the error to an error reporting service.
  - Error Boundaries catch: render-phase errors in child components, constructor errors in children, lifecycle method errors. They do NOT catch: errors in event handlers (use regular try/catch), async errors in `useEffect` or Promises, errors in the Error Boundary itself, and errors in server-side rendering.
  - The correct architecture: multiple Error Boundaries at different granularity levels — a top-level boundary for catastrophic failures, component-level boundaries around isolated features (so one broken widget doesn't crash the entire app).
  - On Apple's Offers Platform, an Error Boundary around the offers grid meant a single malformed offer payload couldn't crash the entire page — it showed a fallback for the grid while navigation remained functional.

- **Example:**

  ```jsx
  class OfferGridErrorBoundary extends React.Component {
    state = { hasError: false, error: null };

    static getDerivedStateFromError(error) {
      return { hasError: true, error };
    }

    componentDidCatch(error, info) {
      // Log to Splunk/error tracker
      logErrorToSplunk(error, info.componentStack);
    }

    render() {
      if (this.state.hasError) {
        return <OfferGridFallback onRetry={() => this.setState({ hasError: false })} />;
      }
      return this.props.children;
    }
  }
  ```

- **Technical Terms to Include:** Error Boundary, `getDerivedStateFromError`, `componentDidCatch`, component stack, fallback UI, graceful degradation, error isolation, event handler error.

- **Gotcha:** "Why can't Error Boundaries be functional components?" → `getDerivedStateFromError` and `componentDidCatch` are class lifecycle methods. React has not provided Hook equivalents yet. There's a community proposal (`use()` in React canary), but as of React 18, class components are the only option.

- **Follow-Up:** "How do you retry after an Error Boundary catches?" → Reset the `hasError` state on user action (a retry button calls `setState({ hasError: false })`). Also consider resetting on route changes. They test real-world recovery patterns.

- **Conclusion:** Error Boundaries are the React equivalent of try/catch at the component tree level — granular placement determines whether a single component failure causes a page crash or a localised degradation.

## Performance › Code Splitting and Lazy Loading

- **Question:** How does code splitting work in React, what is the loading cascade problem it creates, and how do you solve it without degrading UX?

- **Answer:**
  - Code splitting is the practice of splitting the JavaScript bundle into smaller chunks that load on demand, rather than shipping the entire application on initial page load. React provides `React.lazy()` + `Suspense` as the native mechanism.
  - `React.lazy()` accepts a function that returns a dynamic `import()` — when the component is first rendered, the dynamic import is triggered, the chunk is fetched from the CDN, and the component renders after it resolves.
  - The loading cascade problem: lazy components trigger fetching during render — if data fetching also happens inside those components via `useEffect`, you get a network waterfall: JS chunk fetch → render → data fetch → render. Total loading time compounds.
  - Solution: route-level code splitting with preloading — prefetch the JS chunk before navigation (on hover or on route transition initiation), so the chunk is warm when the user arrives.
  - On Apple's Offers Platform, feature-based splitting meant the promotions module (heavy) loaded only when users navigated to the promotions section — initial page load dropped measurably.

- **Example:**

  ```jsx
  // Route-level lazy loading with Suspense
  const OffersPage = React.lazy(() => import('./OffersPage'));
  const CheckoutPage = React.lazy(() => import('./CheckoutPage'));

  const App = () => (
    <Router>
      <Suspense fallback={<AppShellLoader />}>
        <Routes>
          <Route path='/offers' element={<OffersPage />} />
          <Route path='/checkout' element={<CheckoutPage />} />
        </Routes>
      </Suspense>
    </Router>
  );

  // Preloading on hover to eliminate cascade
  const NavLink = ({ to, label }) => (
    <Link to={to} onMouseEnter={() => import('./OffersPage')}>
      {label}
    </Link>
  );
  ```

- **Technical Terms to Include:** code splitting, dynamic `import()`, `React.lazy`, chunk, CDN, waterfall problem, preloading, route-based splitting, Webpack magic comments (`/* webpackPrefetch: true */`).

- **Gotcha:** "Does `React.lazy` work with named exports?" → No — `React.lazy` only works with default exports. For named exports, you create a wrapper module that re-exports the named component as a default, or use a dynamic import with destructuring inline.

- **Follow-Up:** "What's the difference between prefetch and preload in the context of code splitting?" → Preload: browser fetches the resource with high priority, same navigation. Prefetch: browser fetches with low priority during idle time, for future navigations. `webpackPrefetch` is the common choice for route chunks.

- **Conclusion:** Code splitting is not about lazy loading for its own sake — it's about aligning chunk delivery with usage patterns, and the real architectural work is eliminating the network waterfalls that naive lazy loading creates.

## Advanced Patterns › Custom Hooks

- **Question:** What makes a custom Hook architecturally significant beyond just being a shared function, and what are the rules that distinguish a valid custom Hook from an ordinary utility function?

- **Answer:**
  - A custom Hook is a function prefixed with `use` that can call other Hooks — it lets you extract and share stateful logic between components without changing the component hierarchy (no HOC wrapper, no render prop nesting).
  - The architectural significance: it enables a clean separation between _what_ a component renders and _how_ it acquires and manages the data it renders — UI logic and behaviour logic become independently testable and reusable.
  - A custom Hook is required (not just recommended) when the shared logic uses `useState`, `useEffect`, `useContext`, or any other built-in Hook — ordinary functions can't call Hooks because of the Rules of Hooks (Hooks must be called at the top level of a React function).
  - Custom Hooks return whatever the consumer needs — state values, setter functions, derived data, event handlers — the caller controls what it uses.
  - On the Offers Platform, `useOfferEligibility(deviceId)` encapsulated the fetch, loading state, error state, and polling interval — every component using it got consistent, tested behaviour without duplicating that logic.

- **Example:**

  ```jsx
  // Custom Hook: encapsulates fetch + loading + error — used across multiple components
  const useOffers = (deviceId) => {
    const [offers, setOffers] = useState([]);
    const [loading, setLoading] = useState(true);
    const [error, setError] = useState(null);

    useEffect(() => {
      let cancelled = false;
      setLoading(true);
      fetchOffersForDevice(deviceId)
        .then((data) => {
          if (!cancelled) {
            setOffers(data);
            setLoading(false);
          }
        })
        .catch((err) => {
          if (!cancelled) {
            setError(err);
            setLoading(false);
          }
        });
      return () => {
        cancelled = true;
      };
    }, [deviceId]);

    return { offers, loading, error };
  };

  // Consumer: clean component, no fetch logic
  const OffersGrid = ({ deviceId }) => {
    const { offers, loading, error } = useOffers(deviceId);
    if (loading) return <Skeleton />;
    if (error) return <ErrorState />;
    return offers.map((o) => <OfferCard key={o.id} offer={o} />);
  };
  ```

- **Technical Terms to Include:** custom Hook, Rules of Hooks, stateful logic extraction, co-location, testability, separation of concerns, `use` prefix convention, composable Hooks.

- **Gotcha:** "Can you call a custom Hook conditionally?" → No — it violates the Rules of Hooks. The `use` prefix marks a function as subject to Hooks rules; conditional calls break the call-order invariant. If you need conditional logic inside a Hook, the condition goes inside the Hook, not at the call site.

- **Follow-Up:** "How do you test a custom Hook?" → `@testing-library/react-hooks` (or `renderHook` from Testing Library v13+) lets you render a Hook in isolation without a host component. You can assert on return values and fire updates. They test whether you can verify behaviour independent of UI.

- **Conclusion:** Custom Hooks are the primary composition primitive in modern React — they replace HOCs and render props for logic reuse while keeping component code focused exclusively on rendering intent.
