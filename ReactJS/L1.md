## Core Philosophy › Virtual DOM

- **Question:** What is the Virtual DOM, and why did React introduce it instead of operating directly on the real DOM?

- **Answer:**
  - The Virtual DOM is an in-memory JavaScript representation of the actual DOM tree — a lightweight copy React maintains to compute what needs to change before touching the browser.
  - Direct DOM manipulation is expensive because every change triggers layout, paint, and composite cycles in the browser rendering pipeline.
  - React diffs the new Virtual DOM tree against the previous snapshot (the diffing algorithm) and computes a minimal set of mutations — a process called reconciliation.
  - Only the computed diff is flushed to the real DOM in a single batch, reducing unnecessary reflows and repaints.
  - This abstraction also enables React to target non-DOM environments (React Native, React Three Fiber) by decoupling component logic from the renderer.

- **Example:**

  ```jsx
  // Without Virtual DOM: every keystroke triggers direct DOM mutation
  document.getElementById('title').innerText = newValue; // immediate reflow

  // With React: state update → new VDOM tree → diff → single DOM patch
  const [title, setTitle] = useState('Offers');
  // React batches this update and patches only the text node that changed
  setTitle('Apple Offers Platform');
  ```

- **Technical Terms to Include:** Virtual DOM, reconciliation, diffing algorithm, fiber, commit phase, reflow, repaint, renderer abstraction.

- **Gotcha:** "If Virtual DOM is faster, why does Svelte not use it and still outperform React in benchmarks?" — The interviewer wants to hear that Virtual DOM is a performance floor, not a ceiling. Svelte compiles away the abstraction; React's VDOM overhead is real for trivial updates.

- **Follow-Up:** "What is the commit phase in React's render cycle?" → It's where React applies the computed diff to the real DOM, runs `useLayoutEffect`, and fires `ref` callbacks. — They ask this to verify you understand that rendering and committing are two distinct phases.

- **Conclusion:** The Virtual DOM is a trade-off — it adds abstraction overhead but eliminates unpredictable direct DOM mutations, giving React a predictable, batchable update model.

## Core Philosophy › Component Model

- **Question:** React is described as component-based. What does that actually mean architecturally, and what problem does it solve that pre-React jQuery-era development had?

- **Answer:**
  - A component is a self-contained, reusable unit that encapsulates its own markup, logic, and optionally style — a co-location principle that maps closely to UI concerns.
  - Pre-React (jQuery era), markup lived in HTML, behaviour in scattered JS files, and state was implicit in the DOM itself — no single source of truth, no encapsulation boundary.
  - Components enforce a clear contract: props in, rendered output out — making each unit independently testable, replaceable, and composable.
  - The component tree creates a declarative render model — you describe _what_ the UI should look like given current state; React handles _how_ to get there.
  - This also enables design systems: a Button component defined once, used everywhere, updated in one place — the core value delivered on Apple's Offers Platform.

- **Example:**

  ```jsx
  // jQuery era: implicit state in DOM, no boundary
  $('#submitBtn').on('click', () => {
    const val = $('#input').val(); // reading from DOM = DOM as state store
    $('#result').text(val);
  });

  // React: explicit state, declarative output, encapsulated
  const OfferCard = ({ offer, onSelect }) => (
    <div className='offer-card' onClick={() => onSelect(offer.id)}>
      <h3>{offer.title}</h3>
      <span>{offer.discount}% off</span>
    </div>
  );
  ```

- **Technical Terms to Include:** component encapsulation, declarative rendering, props contract, co-location, composition over inheritance, single source of truth.

- **Gotcha:** "If components are so great, why do large React apps still become unmaintainable?" — They want to hear: components solve the UI unit problem, but state architecture across components is a separate concern that React does not solve by default. That's where Redux, Zustand, or server state libraries come in.

- **Follow-Up:** "What's the difference between composition and inheritance in React?" → React favours composition — passing children or specialised components as props — over class inheritance. They ask this to confirm you understand why React deliberately avoided deep class hierarchies.

- **Conclusion:** The component model is React's core architectural promise — UI as composable, declarative, encapsulated functions with an explicit input/output contract.

## Core Philosophy › Unidirectional Data Flow

- **Question:** What is unidirectional data flow in React, and why is it architecturally significant for large-scale applications?

- **Answer:**
  - In React, data flows in one direction: parent → child via props. A child cannot directly mutate its parent's state — it can only notify via callbacks.
  - This makes state changes traceable: if a bug appears, you follow the data flow upward to find the single owner of that state.
  - In bidirectional systems (classic Angular two-way binding), a change anywhere can propagate in multiple directions — making debugging combinatorially harder as the app scales.
  - Unidirectional flow maps cleanly to event sourcing patterns: UI events → dispatch action → state update → re-render. Redux formalises this exact loop.
  - On Apple's Offers Platform, this pattern meant that offer selection state was owned at the layout level and passed down — no child component could produce phantom state mutations.

- **Example:**

  ```jsx
  // Unidirectional: parent owns state, child receives + notifies
  const OffersPage = () => {
    const [selectedOffer, setSelectedOffer] = useState(null);
    return (
      <OfferList
        offers={offers}
        onSelect={setSelectedOffer} // child calls this, parent updates
      />
    );
  };

  // Anti-pattern: child mutating parent ref directly — breaks traceability
  const BadChild = ({ stateRef }) => {
    stateRef.current = 'mutated'; // invisible side effect, untraceable
  };
  ```

- **Technical Terms to Include:** unidirectional data flow, prop drilling, lifting state, single source of truth, event sourcing, derived state, controlled component.

- **Gotcha:** "Doesn't Context API break unidirectional flow?" — No. Context still flows top-down from a Provider. It shortcuts prop drilling but doesn't introduce bidirectional mutation — the Gotcha is distinguishing flow direction from delivery mechanism.

- **Follow-Up:** "When does unidirectional flow become a liability?" → When deeply nested components need to update shared state — prop drilling becomes painful. Answer: lift state, use Context, or move to a state manager. They're testing whether you know the limits of the pattern.

- **Conclusion:** Unidirectional data flow makes state changes predictable and traceable — the single most important architectural property for debugging large-scale React applications.

## Core Philosophy › Reconciliation

- **Question:** Explain React's reconciliation algorithm. What heuristics does it use, and what are the implications of those heuristics for developers?

- **Answer:**
  - Reconciliation is the process by which React computes the diff between the previous and next Virtual DOM trees to determine the minimum set of DOM mutations needed.
  - React uses two key heuristics: (1) elements of different types produce entirely different trees — React tears down and rebuilds rather than diffing deeply; (2) `key` props signal stable identity across renders — React uses them to match list items between renders.
  - Without `key`, React falls back to index-based matching, which breaks when items are reordered — causing state leakage between visually different items.
  - The O(n) reconciliation algorithm (vs the theoretical O(n³) for tree diffing) only works because of these heuristics — violating them degrades performance and correctness simultaneously.
  - React Fiber, introduced in React 16, made reconciliation interruptible — high-priority updates (user input) can preempt low-priority ones (background data fetches).

- **Example:**

  ```jsx
  // Bad: index as key — React reuses DOM nodes incorrectly on reorder
  offers.map((offer, i) => <OfferCard key={i} offer={offer} />);

  // Good: stable unique ID — React correctly identifies each item
  offers.map((offer) => <OfferCard key={offer.id} offer={offer} />);

  // Type heuristic: switching type tears down the tree
  // Before: <OfferCard /> → After: <PromoBanner /> = full unmount + remount
  ```

- **Technical Terms to Include:** reconciliation, diffing algorithm, key prop, stable identity, Fiber, work-in-progress tree, interruptible rendering, O(n) heuristic.

- **Gotcha:** "Can you use array index as key if the list never reorders?" — Technically yes, but it's a fragile assumption. The interviewer wants to hear: use stable IDs by default; index-as-key is a footgun waiting for a future product requirement to reorder the list.

- **Follow-Up:** "What is the work-in-progress tree in Fiber?" → It's the in-memory clone of the current tree that React builds during reconciliation. Only after it's complete does React swap it in during the commit phase. They're probing depth of Fiber knowledge.

- **Conclusion:** Reconciliation's performance depends entirely on the heuristics developers honour — correct `key` usage and consistent element types are not optional niceties, they're load-bearing architectural contracts.

## Core Philosophy › Declarative vs Imperative

- **Question:** React is declarative. Explain what that means, the mental model it produces, and where imperative code still legitimately appears in a React codebase.

- **Answer:**
  - Declarative: you describe the desired UI state given current data — React figures out how to achieve it. Imperative: you describe the steps to manipulate the UI directly.
  - The declarative model inverts control — the developer describes _what_, the framework handles _how_. This eliminates entire classes of bugs around sequencing and state drift.
  - The mental model becomes: `UI = f(state)` — the UI is a pure function of the current state snapshot. Any given state should always produce the same UI.
  - Imperative code legitimately appears when you need to escape React's model: direct DOM access via `useRef`, third-party library integration (D3 chart mutations, video players), focus management, and animations.
  - On PayPal's Checkout, form validation and accessibility focus management required imperative `ref`-based code co-existing with declarative component trees.

- **Example:**

  ```jsx
  // Declarative: describe what the UI should look like
  const SubmitButton = ({ isLoading }) => <button disabled={isLoading}>{isLoading ? 'Processing...' : 'Pay Now'}</button>;

  // Legitimate imperative escape hatch: programmatic focus after modal open
  const Modal = ({ onClose }) => {
    const closeRef = useRef(null);
    useEffect(() => {
      closeRef.current?.focus(); // imperative DOM command, necessary for a11y
    }, []);
    return (
      <button ref={closeRef} onClick={onClose}>
        Close
      </button>
    );
  };
  ```

- **Technical Terms to Include:** declarative rendering, `UI = f(state)`, escape hatch, `useRef`, imperative handle, side effect, controlled abstraction.

- **Gotcha:** "Is `useEffect` declarative or imperative?" — It's a declarative side effect declaration (you declare _when_ it runs via dependencies), but the code inside it can be imperative. Interviewers use this to probe whether you understand the distinction clearly.

- **Follow-Up:** "What's `useImperativeHandle` and when would you use it?" → It exposes a custom imperative API from a child component to a parent via `ref`. Used for focus management, scroll control, media playback from parent. They ask to test advanced API knowledge.

- **Conclusion:** React's declarative model eliminates state-drift bugs by making UI a predictable function of state, but every non-trivial application contains imperative escape hatches — the skill is knowing exactly where to draw that boundary.
