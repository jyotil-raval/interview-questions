## Architecture › Micro-Frontend Integration with React

- **Question:** Walk through how you would architect a React micro-frontend system using Module Federation. What are the hard problems you'd face at the seam between micro-frontends?

- **Answer:**
  - Module Federation (Webpack 5) allows multiple independently deployed applications to share JavaScript modules at runtime — each micro-frontend (MFE) exposes components or modules; the shell application (host) imports them on demand without knowing their internal implementation.
  - The core architecture: Shell (host) → loads remote entries from each MFE → consumes exposed components. Each MFE is independently buildable, deployable, and owns its domain.
  - Hard problem 1 — Shared dependencies: if Shell bundles React 18.2.0 and MFE-A bundles React 18.1.0, you risk two React instances — broken Hooks, broken Context, broken refs. Shared scope configuration in Module Federation resolves this via singleton enforcement.
  - Hard problem 2 — Cross-MFE state: each MFE should own its domain state. Cross-domain state sharing should go through events (custom DOM events, a shared event bus) or URL state — never through direct React Context crossing MFE boundaries.
  - Hard problem 3 — Contract testing: an MFE's exposed API is a contract. A breaking change in an MFE's exposed component signature can silently break the shell at runtime, not at build time. Contract tests (Pact, or custom schema validation) are the only safety net.

- **Example:**

  ```js
  // webpack.config.js — MFE: Apple Offers Grid exposes its root component
  new ModuleFederationPlugin({
    name: 'offersApp',
    filename: 'remoteEntry.js',
    exposes: {
      './OffersGrid': './src/OffersGrid',
      './useOfferEligibility': './src/hooks/useOfferEligibility'
    },
    shared: {
      'react': { singleton: true, requiredVersion: '^18.0.0' },
      'react-dom': { singleton: true, requiredVersion: '^18.0.0' }
    }
  });

  // Shell: lazy loads OffersGrid from remote
  const OffersGrid = React.lazy(() => import('offersApp/OffersGrid'));
  ```

- **Technical Terms to Include:** Module Federation, remote entry, shared scope, singleton dependency, runtime integration, bundle isolation, contract testing, event bus, shell application, federated module.

- **Gotcha:** "What happens if the remote entry fails to load at runtime?" → The shell throws and the Suspense boundary or Error Boundary catches it. You need both a Suspense fallback (loading the remote) and an Error Boundary (network failure or version mismatch). Without both, a single MFE outage crashes the shell.

- **Follow-Up:** "How do you handle authentication across micro-frontends?" → Authentication state must live in the shell (or a shared auth service) and be distributed via URL parameters, custom events, or a shared singleton. Never distribute auth state through module imports — creates tight coupling. They test cross-cutting concern architecture.

- **Conclusion:** Micro-frontends solve organisational scale — independent deployment and ownership — but they transfer the complexity to the integration seam, where shared dependencies, cross-boundary state, and contract stability are the architectural load-bearing concerns.

## Architecture › Design Systems in React

- **Question:** How do you architect a React component library intended as a Design System for consumption by multiple product teams? What are the decisions that determine whether it succeeds or becomes abandoned?

- **Answer:**
  - A Design System is a versioned, distributed library of components that encapsulate design decisions (tokens, spacing, typography) and interaction patterns — the goal is consistency and velocity, not just code reuse.
  - Architecture decisions: (1) Token layer — design tokens (colour, spacing, type scales) as CSS variables or JS constants. Change a token, every consuming component updates. (2) Component API design — props surface must be intentional; every prop is a future maintenance commitment. Composition over configuration.
  - Versioning and breaking changes are the primary failure mode: a team upgrades the design system, a breaking prop change lands, their CI fails — now they distrust the system and pin to an old version. Semantic versioning + automated visual regression tests are non-negotiable.
  - Adoption drivers: quality documentation with live examples (Storybook), zero-config integration, and a clear upgrade path. If a team needs two hours to upgrade, they won't upgrade.
  - On Apple's Offers Platform, the reusable component library reduced feature delivery time because teams composed from audited, accessible, tested primitives rather than rebuilding layout and form elements per feature.

- **Example:**

  ```tsx
  // Token-driven design system: change token → all components update
  // tokens.ts
  export const tokens = {
    color: { primary: 'var(--color-primary)', danger: 'var(--color-danger)' },
    spacing: { sm: '8px', md: '16px', lg: '24px' }
  } as const;

  // Component: consumes tokens, exposes intentional API
  interface ButtonProps {
    variant: 'primary' | 'secondary' | 'destructive';
    size: 'sm' | 'md' | 'lg';
    disabled?: boolean;
    onClick: () => void;
    children: React.ReactNode;
  }
  const Button: React.FC<ButtonProps> = ({ variant, size, ...rest }) => <button className={`btn btn--${variant} btn--${size}`} {...rest} />;
  ```

- **Technical Terms to Include:** design tokens, component API surface, Storybook, visual regression testing (Chromatic/Percy), semantic versioning, peer dependencies, tree shaking, accessibility (WCAG), compound component pattern.

- **Gotcha:** "What's the difference between a component library and a design system?" → A component library is code. A design system is the library plus the design language (tokens, guidelines, principles), documentation, governance process, and adoption support. Teams that ship just the library wonder why it doesn't get adopted.

- **Follow-Up:** "How do you handle theming in a Design System used by multiple brands?" → CSS custom properties (variables) per theme scope, or a token generation layer that outputs theme-specific tokens. CSS-in-JS solutions like styled-components use ThemeProvider. The choice has bundle size and SSR implications.

- **Conclusion:** A Design System's technical architecture is the easy part — the hard part is API stability, documentation quality, and the governance process that makes teams trust it enough to depend on it.

## Architecture › State Architecture at Scale

- **Question:** For a large React application with 50+ components, walk through how you decide where state lives and what signals indicate the current state architecture is breaking down.

- **Answer:**
  - State ownership follows proximity: state lives as close to where it's used as possible. Local component state for UI-only concerns (open/closed, hover, focus). Lifted state for shared sibling concerns. Global state only for genuinely cross-cutting data.
  - The three categories: (1) Server state — remote data, async, cached, possibly stale. Handled by React Query / SWR, not Redux. (2) Client UI state — modal open, selected tab, form draft. Stays local or lifts minimally. (3) Global app state — auth, permissions, feature flags, active user session. Redux, Zustand, or Context.
  - Breakdown signals: prop drilling beyond 3 layers; components importing state from unrelated modules; `useEffect` chains where one state update triggers another via effects (a state machine implemented with side effects is a breakdown sign); Redux store growing into a catch-all for everything.
  - Separation principle: don't put server state in Redux. Redux is for client state; React Query is for server state. Mixing them produces dual caches with synchronisation logic that fights itself.
  - On Altimetrik/PayPal: billing subscription status was server state (React Query), checkout session was client global state (Redux), payment form fields were local controlled state — clear separation that survived 18 months of feature additions.

- **Example:**

  ```tsx
  // Correct separation: server state vs client state
  const CheckoutPage = () => {
    // Server state: React Query owns cache, staleness, refetch
    const { data: paymentMethods } = useQuery(['paymentMethods'], fetchPaymentMethods);

    // Client UI state: local, no global needed
    const [selectedMethodId, setSelectedMethodId] = useState(null);

    // Global client state: Redux — persists across pages
    const cartTotal = useSelector(state => state.cart.total);

    return (/* ... */);
  };
  ```

- **Technical Terms to Include:** state proximity, server state, client state, React Query, SWR, Redux, Zustand, cache invalidation, stale-while-revalidate, selector, normalisation, state machine.

- **Gotcha:** "Why shouldn't you put API response data in Redux?" → Redux is synchronous, serialisable client state. API data is asynchronous, cached, potentially stale, and needs invalidation logic — Redux handles this poorly. React Query builds its entire model around these requirements. Mixing them means you maintain two caches.

- **Follow-Up:** "What is Zustand and when would you reach for it over Redux?" → Zustand is a minimal state manager — no boilerplate, direct mutation via immer, selective subscription. Reach for it when Redux's action/reducer ceremony is overkill but you need state outside React Context. They test awareness of the current state management landscape.

- **Conclusion:** State architecture at scale is about categorical discipline — server state, client UI state, and global app state each have different tools, and mixing their ownership produces the synchronisation bugs that scale into 3am incidents.

## Architecture › Server-Side Rendering with React

- **Question:** What problems does SSR solve in a React application, what does it cost, and how do hydration errors surface at runtime?

- **Answer:**
  - SSR solves two distinct problems: (1) Time-to-first-contentful-paint (TTFCP) — the server renders HTML immediately; the user sees content before React loads and hydrates. (2) SEO — crawlers receive fully rendered HTML without waiting for JavaScript execution.
  - The server renders the component tree to an HTML string (`renderToString` or streaming via `renderToPipeableStream`). The browser receives this HTML, renders it immediately, then React "hydrates" — attaches event listeners and takes over.
  - Cost: (1) Server CPU — every request triggers a render, adding latency and compute cost. (2) Complexity — code must run in both Node.js and browser environments; browser-only APIs (`window`, `document`) must be guarded. (3) State serialisation — initial data fetched on the server must be serialised into the HTML for the client to rehydrate without a second fetch.
  - Hydration errors occur when the server-rendered HTML doesn't match what React would render on the client — a mismatch causes React to discard the server HTML and re-render from scratch (full client render, defeating SSR's purpose).
  - Common mismatch sources: date/time rendering, random IDs, browser-specific data (`localStorage`), or conditional rendering that behaves differently in Node vs browser.

- **Example:**

  ```tsx
  // Next.js: getServerSideProps fetches data, serialises into page props
  export const getServerSideProps = async ({ params }) => {
    const offers = await fetchOffersForDevice(params.deviceId);
    return { props: { offers } }; // serialised into <script> tag in HTML
  };

  // Hydration error source: non-deterministic render
  const BadComponent = () => (
    <div>{Math.random()}</div> // server value ≠ client value = hydration mismatch
  );

  // Fix: suppress or defer client-only rendering
  const SafeComponent = () => {
    const [mounted, setMounted] = useState(false);
    useEffect(() => setMounted(true), []);
    return mounted ? <div>{Math.random()}</div> : null;
  };
  ```

- **Technical Terms to Include:** SSR, hydration, `renderToString`, `renderToPipeableStream`, TTFCP, SEO, hydration mismatch, dehydration/rehydration (React Query), streaming SSR, React Server Components.

- **Gotcha:** "What's the difference between SSR and React Server Components?" → SSR renders the full component tree on the server and sends HTML + JS for hydration. React Server Components never ship their JS to the client — they render to a serialised format on the server with zero client-side JS cost. Fundamentally different execution model.

- **Follow-Up:** "What is streaming SSR and what does it change?" → Instead of waiting for the entire page to render before sending HTML (`renderToString`), streaming sends HTML chunks as they're ready (`renderToPipeableStream`). Slow data sources don't block fast-rendering parts. They test Next.js / React 18 architectural awareness.

- **Conclusion:** SSR is the right trade-off when TTFCP and SEO requirements outweigh server compute cost and code complexity — but hydration correctness is a strict contract, and any non-determinism between server and client renders is a silent correctness failure.

## Architecture › Performance at Scale (Virtualisation and Large Lists)

- **Question:** How would you architect a React UI that needs to display and interact with a list of 10,000 items without degrading to an unusable render performance?

- **Answer:**
  - Rendering 10,000 DOM nodes is the direct cause of the problem — browser layout cost scales with DOM size regardless of how optimised the React rendering is. The solution is virtualisation: render only the DOM nodes currently visible in the viewport.
  - Windowing libraries (react-window, react-virtual, TanStack Virtual) maintain a "window" of rendered items — as the user scrolls, items entering the viewport are rendered and items leaving are unmounted. The DOM stays at ~20–50 nodes regardless of list size.
  - Architecture considerations: (1) Item height must be known or measurable — variable height items require dynamic size measurement (`useVirtualizer` in TanStack handles this). (2) Absolute positioning — virtualised items are positioned via `transform: translateY` or `top` based on their calculated scroll offset. (3) Overscan — rendering a few items above and below the visible window prevents flash-of-empty-content on fast scrolls.
  - Additional layer: if the 10,000 items come from the server, pagination or infinite scroll (React Query's `useInfiniteQuery`) must work together with virtualisation — only fetching pages the user has scrolled to.
  - On IoT dashboards at Volansys, device telemetry tables displayed 5,000+ sensor readings — react-window reduced initial render time from 4+ seconds to under 200ms.

- **Example:**

  ```tsx
  import { useVirtualizer } from '@tanstack/react-virtual';

  const DeviceList = ({ devices }) => {
    const parentRef = useRef(null);
    const virtualizer = useVirtualizer({
      count: devices.length,
      getScrollElement: () => parentRef.current,
      estimateSize: () => 64, // estimated row height in px
      overscan: 5
    });

    return (
      <div ref={parentRef} style={{ height: '600px', overflow: 'auto' }}>
        <div style={{ height: `${virtualizer.getTotalSize()}px`, position: 'relative' }}>
          {virtualizer.getVirtualItems().map((virtualItem) => (
            <div
              key={virtualItem.key}
              style={{
                position: 'absolute',
                top: `${virtualItem.start}px`,
                width: '100%'
              }}>
              <DeviceRow device={devices[virtualItem.index]} />
            </div>
          ))}
        </div>
      </div>
    );
  };
  ```

- **Technical Terms to Include:** virtualisation, windowing, react-window, TanStack Virtual, overscan, scroll offset, DOM node count, infinite scroll, `useInfiniteQuery`, layout thrashing, `requestAnimationFrame`.

- **Gotcha:** "Does virtualisation work with dynamic/variable row heights?" → Yes, but it's harder. `react-window` requires you to pre-calculate heights. TanStack Virtual's `measureElement` callback measures DOM nodes after render and recalculates positions dynamically. They test whether you know the complexity trade-off.

- **Follow-Up:** "When would you choose pagination over virtualisation?" → When deep-linking to specific items matters (users need to share URLs to specific positions), when accessibility requirements make infinite scroll problematic, or when the dataset is small enough that pagination is simpler. They want trade-off thinking.

- **Conclusion:** Large list performance is not a React optimisation problem — it's a DOM node count problem, and virtualisation solves it by decoupling the list size from the rendered DOM size.
