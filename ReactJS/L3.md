## State Management › Context API

- **Question:** What is the Context API, when should you use it, and what are the performance trade-offs that make it the wrong choice for high-frequency state?

- **Answer:**
  - Context provides a mechanism to make a value available to any component in the tree without explicit prop drilling — the Provider broadcasts a value; any Consumer or `useContext` call subscribes to it.
  - When the Context value changes, every component that calls `useContext` for that context re-renders, regardless of whether the specific slice it uses changed. There is no selector mechanism built in.
  - This makes Context correct for low-frequency, near-static data: auth state, theme, locale, feature flags — data that changes rarely and is consumed broadly.
  - High-frequency state (form inputs, real-time IoT sensor data, animation state) in Context causes cascade re-renders — every subscriber fires on every keystroke or sensor tick.
  - On the Apple Offers Platform, locale and authentication context worked perfectly; offer selection state was deliberately kept in component-local state to avoid re-rendering the entire offer grid on every selection.

- **Example:**

  ```jsx
  // Correct use: static-ish auth state in Context
  const AuthContext = createContext(null);
  const AuthProvider = ({ children }) => {
    const [user, setUser] = useState(null);
    return <AuthContext.Provider value={{ user, setUser }}>{children}</AuthContext.Provider>;
  };

  // Performance trap: putting form state in Context
  const FormContext = createContext(null); // DON'T: every keystroke re-renders all consumers
  ```

- **Technical Terms to Include:** Context Provider, `useContext`, value propagation, cascade re-render, referential equality, context splitting, selector pattern, `React.memo` + Context boundary.

- **Gotcha:** "How do you optimise Context to avoid unnecessary re-renders?" → Split contexts by update frequency. Memoize the context value with `useMemo`. Use `React.memo` on consumers. Or move to a state library with built-in selectors (Zustand, Redux). They're testing whether you know Context's ceiling.

- **Follow-Up:** "What's the difference between Context and Redux?" → Context is a delivery mechanism; Redux is a state management pattern. Context replaces prop drilling; Redux replaces ad-hoc state management with predictable, debuggable, serialisable state. They can coexist.

- **Conclusion:** Context is the right tool for infrequent, broadly consumed state — using it for high-frequency state is a performance footgun that produces invisible cascade re-renders across the entire subscriber tree.

## Performance › Memoization (React.memo, useMemo, useCallback)

- **Question:** Explain the three memoization primitives in React — `React.memo`, `useMemo`, and `useCallback` — their exact use cases, and the anti-pattern of over-memoizing.

- **Answer:**
  - `React.memo` is a higher-order component that wraps a functional component and skips re-rendering if its props haven't changed (shallow equality by default). It prevents child re-renders caused by parent re-renders where the child's props are unchanged.
  - `useMemo` memoizes the result of an expensive computation — the function only re-runs when its listed dependencies change. Use for costly transformations (filtering/sorting large arrays, deriving complex structures).
  - `useCallback` memoizes a function reference — returns the same function instance across renders unless dependencies change. Necessary when passing callbacks to `React.memo`-wrapped children (because a new function reference on every render would defeat the `React.memo` optimisation).
  - Over-memoizing is a real anti-pattern: every memoization adds memory cost and a dependency comparison cost. For cheap computations and infrequently re-rendered components, memoization adds overhead without measurable benefit.
  - On the Offers Platform, `React.memo` on `OfferCard` prevented the entire grid from re-rendering when only one offer was selected — the specific, measurable gain that justified the memoization.

- **Example:**

  ```jsx
  // useMemo: expensive computation — only recompute when offers list changes
  const eligibleOffers = useMemo(() => offers.filter((o) => o.eligible && o.expiresAt > Date.now()), [offers]);

  // useCallback: stable reference for memoized child
  const handleSelect = useCallback((offerId) => {
    setSelectedOffer(offerId);
  }, []); // no deps — function logic doesn't capture changing values

  // React.memo: skip re-render if props unchanged
  const OfferCard = React.memo(({ offer, onSelect }) => <div onClick={() => onSelect(offer.id)}>{offer.title}</div>);
  ```

- **Technical Terms to Include:** memoization, shallow equality, referential stability, dependency array, render skip, HOC wrapper, memory cost, profiler, premature optimisation.

- **Gotcha:** "`useCallback` always improves performance — true or false?" → False. If the child is not wrapped in `React.memo`, a stable callback reference provides zero render-skip benefit. The pair works together; one without the other is wasted overhead.

- **Follow-Up:** "How do you measure whether memoization is actually helping?" → React DevTools Profiler — flamegraph view shows which components rendered and why. Without measurement, memoization is speculative. They test whether you profile before optimising.

- **Conclusion:** The three memoization primitives are precision tools, not defaults — applied without profiling data, they add complexity and memory overhead in exchange for unmeasured performance gains.

## Patterns › Controlled vs Uncontrolled Components

- **Question:** Contrast controlled and uncontrolled components in React. When does each pattern fit, and what breaks when you mix them?

- **Answer:**
  - Controlled component: React state is the single source of truth for the input's value. Every keystroke fires an `onChange` → state update → re-render cycle. React owns the value.
  - Uncontrolled component: the DOM is the source of truth. A `ref` reads the value on demand (on submit, on blur). React does not manage the value between interactions.
  - Controlled fits: complex validation on every keystroke, conditional rendering based on field values, multi-step forms with interdependencies, form state that affects other parts of the UI.
  - Uncontrolled fits: simple file inputs (DOM-only by spec), performance-critical high-frequency inputs, integrating third-party non-React inputs, forms where validation only runs on submit.
  - Mixing them (starting controlled, switching to uncontrolled by passing `undefined` as value) triggers React's warning: "A component is changing a controlled input to be uncontrolled" — and the behaviour becomes unpredictable.

- **Example:**

  ```jsx
  // Controlled: React owns value — PayPal CVV field with real-time validation
  const CVVInput = () => {
    const [cvv, setCvv] = useState('');
    const isValid = /^\d{3,4}$/.test(cvv);
    return (
      <>
        <input value={cvv} onChange={(e) => setCvv(e.target.value)} maxLength={4} />
        {!isValid && cvv.length > 0 && <span>Invalid CVV</span>}
      </>
    );
  };

  // Uncontrolled: DOM owns value — file upload
  const FileUpload = () => {
    const fileRef = useRef(null);
    const handleSubmit = () => console.log(fileRef.current.files[0]);
    return <input type='file' ref={fileRef} onChange={handleSubmit} />;
  };
  ```

- **Technical Terms to Include:** controlled input, uncontrolled input, single source of truth, `defaultValue`, `ref`, `onChange` handler, value prop, DOM-owned state.

- **Gotcha:** "Why can't you use `value` and `defaultValue` together on the same input?" → `value` makes it controlled; `defaultValue` sets the initial DOM value for uncontrolled. Using both sends contradictory signals to React — it will prioritise `value` and `defaultValue` is silently ignored.

- **Follow-Up:** "React Hook Form uses uncontrolled components — why?" → It avoids re-rendering the form on every keystroke by not coupling field values to state. This is a deliberate performance choice. They test awareness of library trade-offs.

- **Conclusion:** Controlled inputs give you real-time state visibility at the cost of a re-render per keystroke; uncontrolled inputs defer to the DOM for performance but sacrifice real-time state access — the choice is a deliberate trade-off, not a default.

## Patterns › Higher-Order Components (HOCs)

- **Question:** What is a Higher-Order Component, what problem does it solve, and why has the React community largely migrated away from it toward Hooks?

- **Answer:**
  - A Higher-Order Component is a function that takes a component and returns a new component — it wraps the input component to inject behaviour, props, or context without modifying the original.
  - HOCs solved cross-cutting concerns before Hooks: authentication guards, analytics tracking, theming, data fetching — logic that multiple components needed but shouldn't contain directly.
  - Problems with HOCs: (1) Wrapper hell — deeply nested HOCs in DevTools make component trees unreadable; (2) Prop collision — the HOC injects a prop that conflicts with the wrapped component's own prop; (3) Static composition — HOCs wire behaviour at definition time, not render time.
  - Hooks solve the same cross-cutting concerns with a flat composition model — `useAuth()`, `useAnalytics()`, `useTheme()` are called inline, don't create wrapper components, and don't risk prop collisions.
  - HOCs are still valid in specific scenarios: wrapping class components (which can't use Hooks), third-party library integrations like React Router's `withRouter`, and code that needs to operate at the component identity level (not just logic).

- **Example:**

  ```jsx
  // HOC: inject auth check — creates wrapper layer
  const withAuth = (WrappedComponent) => (props) => {
    const { user } = useAuth();
    if (!user) return <Redirect to='/login' />;
    return <WrappedComponent {...props} user={user} />;
  };
  const ProtectedOffers = withAuth(OffersPage);

  // Hook equivalent: flat, no wrapper, no prop injection risk
  const OffersPage = () => {
    const { user } = useAuth(); // same logic, no HOC layer
    if (!user) return <Redirect to='/login' />;
    return <div>...</div>;
  };
  ```

- **Technical Terms to Include:** higher-order component, cross-cutting concern, wrapper hell, prop collision, static composition, `displayName`, `forwardRef`, `React.memo` + HOC interaction.

- **Gotcha:** "What breaks when you use `React.memo` inside a HOC?" → The HOC creates a new component reference on every render unless memoized. You need to apply `React.memo` to the HOC result, not the inner component, and set `displayName` explicitly for DevTools legibility.

- **Follow-Up:** "What is `forwardRef` and why is it needed with HOCs?" → HOCs wrap components and break `ref` forwarding — `React.forwardRef` explicitly passes refs through the HOC to the underlying DOM element. They test HOC + ref knowledge together.

- **Conclusion:** HOCs are a solved pattern — Hooks supersede them for logic reuse in functional components, but HOCs remain the only option for class component cross-cutting concerns and cases requiring component-level wrapping.

## Patterns › Render Props

- **Question:** Explain the render props pattern, its use case, and how it compares to HOCs and Hooks for logic reuse.

- **Answer:**
  - Render props is a pattern where a component exposes its internal state or behaviour to the caller by calling a function prop — typically named `render` or `children` — with that data as arguments.
  - It solves the same cross-cutting concern problem as HOCs: sharing logic (mouse position, toggle state, data fetching status) across components without lifting that logic to each consumer.
  - Advantage over HOCs: composition is explicit at render time — you can see exactly what data flows where in the JSX. No prop collision risk because the consumer destructures what it needs.
  - Disadvantage: callback nesting creates "pyramid of doom" for multiple shared concerns — wrapping three render prop providers creates three levels of nesting. HOCs have the same problem but it's hidden.
  - Hooks completely replace render props for logic reuse in functional components — `useMouse()`, `useToggle()`, `useDataFetch()` — flat, composable, no JSX nesting required.

- **Example:**

  ```jsx
  // Render props: DataFetcher exposes data to caller
  const DataFetcher = ({ url, children }) => {
    const [data, setData] = useState(null);
    const [loading, setLoading] = useState(true);
    useEffect(() => {
      fetch(url)
        .then((r) => r.json())
        .then((d) => {
          setData(d);
          setLoading(false);
        });
    }, [url]);
    return children({ data, loading }); // calls the render function
  };

  // Consumer: explicit about what it receives
  <DataFetcher url='/api/offers'>{({ data, loading }) => (loading ? <Spinner /> : <OfferList offers={data} />)}</DataFetcher>;
  ```

- **Technical Terms to Include:** render prop, function-as-child, inversion of control, callback hell, logic encapsulation, slot pattern, children prop.

- **Gotcha:** "Is the `children` prop a render prop?" → Only if the component calls `children` as a function — `children({ data })`. If it renders `children` directly as a React node, it's the composition/slot pattern, not render props.

- **Follow-Up:** "When would you still use render props over Hooks today?" → When you need to share rendering logic between class and functional components, or when you're building a headless UI library where the caller controls the rendering entirely (e.g., Downshift, React Table v7 and below).

- **Conclusion:** Render props made cross-cutting logic reuse explicit at the cost of JSX nesting depth — Hooks have superseded them in functional codebases, but the pattern remains relevant for headless UI libraries that intentionally invert rendering control.
